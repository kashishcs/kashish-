Background

About

I approach computer science as a structured discipline built on fundamental principles rather than isolated techniques. My work emphasizes clarity, logical progression, and systematic problem decomposition.

Rather than seeking shortcuts or surface-level implementations, I invest time in understanding core mechanisms—how algorithms function at their foundation, why certain data structures suit specific contexts, and how object-oriented principles enable maintainable system architecture. This deliberate approach has shaped both my learning methodology and my engineering philosophy.

Technical Philosophy

I believe that effective software engineering begins with disciplined thinking. Before writing code, I prioritize understanding the problem space completely—identifying constraints, edge cases, and potential failure modes. This upfront investment in comprehension consistently yields more robust solutions than reactive debugging.

My preference for strongly typed languages like Java stems from their enforcement of explicit structure. The compilation process serves as an early verification layer, catching type mismatches and contract violations before runtime. This aligns with my broader belief that constraints, when thoughtfully applied, enhance rather than limit creative problem-solving.

I view algorithms not as memorized recipes but as compositions of fundamental operations—comparisons, assignments, iterations, and recursions. By deconstructing complex algorithms into these atomic components, I can reason about time complexity, space requirements, and correctness proofs with greater precision. This granular understanding proves essential when adapting standard algorithms to domain-specific requirements.

Learning Approach

My learning process follows a deliberate cycle: study foundational concepts through authoritative sources, implement them in isolated environments to verify comprehension, then apply them within larger system contexts to understand integration challenges. This progression from theory to practice to application has proven more durable than alternative approaches.

When encountering new data structures or algorithmic paradigms, I resist the temptation to immediately utilize library implementations. Instead, I first build minimal versions from scratch—implementing a binary search tree manually, coding sorting algorithms without external dependencies, or constructing graph traversal logic using only primitive data types. This hands-on construction phase solidifies understanding in ways that passive consumption cannot achieve.

I maintain detailed documentation of my learning process—not as polished artifacts, but as working notes that capture decision points, alternative approaches considered, and lessons extracted from failures. These records serve dual purposes: they externalize my reasoning for future review, and they train me to articulate technical decisions clearly, a skill essential for collaborative engineering environments.

Technical Mindset

I approach debugging as an investigative discipline. Rather than applying random fixes or immediately consulting external resources, I first instrument code with strategic logging, reproduce failures consistently, and form hypotheses about root causes. This systematic debugging methodology often reveals deeper issues than the surface symptom initially suggested.

Performance optimization, in my view, requires measurement before modification. I've learned to profile code empirically rather than optimizing based on intuition, as actual bottlenecks frequently diverge from predicted ones. This data-driven approach extends to algorithm selection—choosing appropriate structures based on access patterns and scale requirements rather than theoretical ideals.

Code readability represents a form of technical communication. I structure programs to minimize cognitive load for future readers—preferring explicit variable names over abbreviations, decomposing complex expressions into intermediate steps, and organizing logic to match conceptual flow. Well-architected code should read as a clear explanation of its own purpose, reducing the maintenance burden across a project's lifecycle.

Ultimately, I view computer science as an intellectual discipline that demands rigor, patience, and continuous refinement. Each project serves as both a practical deliverable and a learning artifact—an opportunity to strengthen fundamental skills while producing functional systems. This dual perspective on work product ensures sustained growth alongside tangible output.

01

Algorithm Visual Playground

Educational Tool 2025

Overview

An interactive platform for visualizing fundamental algorithms through step-by-step execution. The system renders algorithm behavior as animated graphical representations, making abstract computational processes tangible and observable. Users can control execution speed, step through operations individually, and observe state changes at each iteration.

The project emerged from a recognition that algorithm comprehension deepens significantly when students can observe intermediate states rather than only initial inputs and final outputs. By externalizing the execution model, the visualizer transforms algorithm study from passive reading into active investigation.

Problem Context

Traditional algorithm education relies heavily on static code listings and textual descriptions of execution flow. While these resources convey the mechanics of an algorithm, they fail to illustrate the dynamic transformations that occur during execution. Students often struggle to internalize how data structures evolve across iterations, particularly for recursive algorithms where call stack visualization proves essential for comprehension.

Existing visualization tools frequently prioritize visual polish over pedagogical clarity, introducing unnecessary complexity that obscures the underlying algorithm. This project aimed to create a minimal, distraction-free environment where the algorithm itself remains the primary focus, with visualizations serving as explanatory aids rather than decorative elements.

Implementation Approach

The architecture separates algorithm logic from visualization rendering through a state management layer. Each algorithm implements a generator function that yields state snapshots at significant operations—comparisons, swaps, recursive calls, or data structure modifications. The visualization engine consumes these state snapshots and renders corresponding graphical updates.

This separation ensures that algorithm implementations remain clean and readable, mirroring textbook pseudocode closely. The generator pattern allows algorithms to execute step-by-step without complex threading or callback structures, providing natural breakpoints for visualization updates. State snapshots include not only data values but also metadata about the current operation, enabling contextual annotations in the visual display.

The rendering system employs canvas-based drawing for performance, particularly important when visualizing large datasets or rapid state changes. Transitions between states use interpolation to create smooth animations that guide the viewer's attention to relevant changes rather than forcing them to identify differences manually.

Algorithms Covered

  • Sorting Algorithms: Implementations of bubble sort, selection sort, insertion sort, merge sort, quick sort, and heap sort. Each visualization highlights comparison operations, element movements, and partition boundaries. The merge sort visualization explicitly shows the divide and conquer process through tree representation.
  • Recursion Patterns: Visual call stack representation for recursive functions including factorial calculation, Fibonacci sequence generation, and Tower of Hanoi solution. The stack view shows activation records, parameter values, and return flow.
  • Graph Traversal: Breadth-first search and depth-first search visualized on configurable graph structures. The system animates frontier expansion, visited node tracking, and parent pointer updates used for path reconstruction.
  • Pathfinding: Implementation of Dijkstra's algorithm and A* search with weighted graphs. Visualization includes priority queue state, distance label updates, and final path highlighting once the target node is reached.

Design Focus

The interface prioritizes clarity through minimalism. Controls are limited to essential functions: play, pause, step forward, step backward, and speed adjustment. Algorithm selection uses a sidebar menu organized by category. The visualization area occupies maximum screen space, avoiding unnecessary chrome or decorative elements.

Color coding follows consistent semantic meaning across all visualizations: elements currently being compared appear in one color, elements being moved in another, and already-sorted or visited elements in a third. This consistency reduces cognitive load when switching between different algorithms, as users can apply learned visual vocabulary universally.

Annotations appear contextually during execution, explaining what operation is currently occurring without requiring users to consult separate documentation. These annotations are brief—typically a single sentence—and positioned adjacent to the relevant visual elements.

Technical Stack

  • Python for algorithm implementations and state generation logic
  • JavaScript for visualization engine and user interaction handling
  • HTML5 Canvas API for rendering performance-critical animations
  • Custom data structures including priority queues, adjacency lists, and tree representations

Plagiarism Detection System

Text Analysis 2025

Overview

A text comparison system that identifies potential plagiarism through multiple similarity detection algorithms. The tool accepts two text documents as input and produces a detailed similarity report including overall similarity scores, matching segments, and algorithm-specific analysis. The system demonstrates practical application of string algorithms, set operations, and statistical text processing techniques.

Rather than relying on a single similarity metric, the system implements multiple complementary approaches that capture different aspects of text similarity. This multi-algorithm design provides more robust detection than any single method while also serving as an educational artifact that compares different algorithmic strategies for the same problem domain.

Text Preprocessing Pipeline

Before similarity analysis, both documents undergo normalization to ensure consistent comparison. The preprocessing pipeline converts all text to lowercase, removes punctuation that doesn't contribute to semantic meaning, and normalizes whitespace. Optionally, the system can apply stemming to reduce words to their root forms, treating variations like "running," "runs," and "ran" as equivalent.

This preprocessing phase is critical because raw text contains significant variability unrelated to actual content similarity—differences in capitalization, formatting, or inflectional word forms. By standardizing these surface-level features, the similarity algorithms can focus on substantive content comparison rather than stylistic variations.

The system preserves the original text alongside the processed version, allowing the final report to highlight matching segments in their original form while performing similarity calculations on normalized text. This dual representation ensures both algorithmic correctness and user-friendly output.

Jaccard Similarity Algorithm

Jaccard similarity treats each document as a set of unique tokens (typically words) and measures overlap as the ratio of shared tokens to total unique tokens across both documents. Formally, Jaccard similarity equals the size of the intersection divided by the size of the union of the two token sets.

This metric excels at detecting documents with substantial vocabulary overlap regardless of word order. It's particularly effective for identifying cases where content has been paraphrased with synonyms but maintains similar overall vocabulary. However, Jaccard similarity ignores token frequency, treating words that appear once identically to words that appear dozens of times.

Implementation uses hash sets for efficient set operations. Token insertion, intersection computation, and union calculation all operate in linear time relative to document size, making Jaccard similarity computationally economical even for large documents.

Cosine Similarity Algorithm

Cosine similarity represents documents as vectors in high-dimensional space where each dimension corresponds to a unique token across both documents. Vector components indicate token frequency—typically using term frequency (raw count) or TF-IDF weighting. Similarity is computed as the cosine of the angle between the two document vectors.

Unlike Jaccard similarity, cosine similarity accounts for token frequency, making it sensitive to documents that not only share vocabulary but use those words with similar emphasis. Documents with proportional word frequency distributions yield high cosine similarity even if absolute frequencies differ, making the metric robust to document length variations.

The implementation constructs sparse vector representations using dictionaries that map tokens to frequencies. The cosine calculation involves computing dot product (sum of componentwise products) and vector magnitudes (square root of sum of squared components), both implementable in single passes through the token frequency maps.

Sequence Matching Algorithm

Sequence matching identifies contiguous sequences of tokens that appear identically in both documents, using a longest common subsequence approach. This algorithm detects copied passages that maintain original word order, even if surrounded by different content.

The implementation uses dynamic programming to build a similarity matrix where cell (i,j) represents the length of the longest common sequence ending at position i in document 1 and position j in document 2. Matching tokens extend the sequence length from the previous diagonal cell; mismatches reset to zero. Maximum values in this matrix indicate the longest shared sequences.

After identifying matching sequences, the system performs backtracking to extract the actual text spans. The final report highlights these sequences in both documents, providing concrete evidence of potential plagiarism rather than abstract similarity scores. This visualization proves particularly valuable for human review of flagged document pairs.

Why Multiple Algorithms

Each algorithm captures different plagiarism patterns. Jaccard similarity excels at detecting rewrites with similar vocabulary. Cosine similarity accounts for emphasis and structure through frequency weighting. Sequence matching catches verbatim copying of passages. By combining all three, the system provides comprehensive analysis that's less susceptible to evasion tactics like synonym substitution or sentence reordering.

The multi-algorithm approach also serves pedagogical value, demonstrating how different mathematical formulations address the same high-level goal through distinct mechanisms. Users can observe how Jaccard and cosine metrics might yield different scores for the same document pair, building intuition about what each metric measures.

System Architecture

The codebase separates concerns into distinct modules: a preprocessing module handles text normalization, individual algorithm modules implement each similarity metric with clear input/output contracts, and a report generation module aggregates results and formats output. This modular architecture simplifies testing, as each algorithm can be validated independently using known test cases.

Optional database integration allows the system to compare submitted documents against a corpus of previously analyzed texts, enabling detection of plagiarism from stored sources rather than just pairwise comparison. The database stores document identifiers and precomputed token sets or vectors, enabling efficient batch comparison without reprocessing the entire corpus for each new submission.

Technical Stack

  • Python for core algorithm implementations and text processing pipelines
  • String algorithms including tokenization, stemming, and normalization
  • Set operations and hash-based data structures for efficient comparison
  • Dynamic programming for sequence matching algorithms
  • Optional SQL integration for corpus-based comparison
  • JavaScript front-end for browser-based usage with file upload interface

Object-Oriented Quiz System

Educational Application 2025

Overview

An interactive quiz application designed as an exercise in object-oriented programming principles. The system manages question banks, tracks user responses, calculates scores, and generates detailed results reports. Beyond its functional purpose, the project serves as a demonstration of class hierarchy design, encapsulation, and separation of concerns in a multi-component application.

The architecture prioritizes extensibility—adding new question types, scoring algorithms, or presentation modes requires minimal modification to existing code. This design philosophy reflects real-world software engineering where requirements evolve and systems must accommodate change without requiring complete rewrites.

Object-Oriented Architecture

The system centers on a class hierarchy rooted in an abstract Question base class. This base class defines the interface that all concrete question types must implement: methods for displaying the question, validating user input, checking correctness, and retrieving point values. Specific question types—multiple choice, true/false, short answer, and multi-select—inherit from this base and provide their own implementations.

This inheritance structure demonstrates the open/closed principle: the Quiz controller class operates on Question references without needing knowledge of specific question types. New question formats can be added by creating additional subclasses without modifying the core quiz logic, ensuring that extensions don't introduce regressions in existing functionality.

A QuestionBank class manages collections of questions, providing methods for random selection, category-based filtering, and difficulty-based sorting. This encapsulation ensures that quiz assembly logic remains separate from individual question behavior, maintaining clean separation of concerns across the codebase.

Question Bank System

Questions are stored in structured data files (JSON format) with schema validation to ensure consistency. Each question entry includes the question text, answer options (for multiple choice types), correct answer(s), point value, difficulty rating, and category tags. This metadata enables sophisticated quiz generation—creating quizzes that target specific topics or difficulty levels.

The QuestionBank class loads these files at initialization, performing validation to catch malformed data before runtime. Questions are parsed into their appropriate subclass instances based on type indicators in the data file. This parsing logic is encapsulated within factory methods that instantiate the correct question subclass, isolating object creation logic from business logic.

For applications requiring dynamic question sets, the system supports database-backed question storage. An abstract DataAdapter interface defines methods for question retrieval and storage, with concrete implementations for both file-based and database-backed persistence. The Quiz system depends only on this abstract interface, allowing transparent substitution of data sources without modifying quiz logic.

Quiz Session Management

A Quiz class orchestrates the overall quiz experience. Upon initialization, it accepts a collection of Question objects and configuration parameters including time limits, whether to show immediate feedback, and scoring mode. The Quiz maintains state tracking the current question index, user responses, elapsed time, and whether the session has completed.

The class provides methods for advancing to the next question, returning to previous questions, submitting answers, and retrieving the current question for display. This interface allows the presentation layer (UI) to remain thin—it need only call these methods and render the returned data, without implementing any quiz logic itself.

Session state can be serialized to enable pause/resume functionality. The Quiz class exports its state as a data structure capturing current position, user responses, and time remaining. This structure can be persisted to browser storage or a database, then rehydrated later to resume the session from the exact point of interruption.

Scoring and Results Logic

Scoring calculation is delegated to a separate ScoreCalculator class that implements different scoring strategies through a strategy pattern. The base strategy awards points for correct answers and zero points for incorrect ones. Alternative strategies implement negative scoring for wrong answers (discouraging guessing) or partial credit for multi-select questions where some but not all correct answers were selected.

Results generation produces detailed reports beyond simple scores. The report includes per-question feedback showing what the user answered, the correct answer, and explanatory text for why that answer is correct. Aggregate statistics show performance by category and difficulty level, helping users identify areas needing additional study.

The reporting system is templatable—a ResultsFormatter abstract class defines the structure of results output, with concrete implementations for plain text, HTML, and JSON formats. This allows the same quiz session to generate results appropriate for terminal display, web rendering, or API responses without duplicating logic.

Design Patterns Applied

Beyond inheritance and polymorphism, the project demonstrates several classical design patterns. The factory pattern encapsulates question instantiation. The strategy pattern enables interchangeable scoring algorithms. The template method pattern structures the quiz flow—the Quiz class defines the overall sequence while delegating specific steps to subclass implementations.

Observer pattern implementation allows external components to subscribe to quiz events—question answered, quiz completed, time warnings. This decoupling means UI components can react to events without the Quiz class maintaining explicit references to UI elements, reducing coupling between the model and view layers.

Educational Focus

This project deliberately emphasizes structure and maintainability over feature richness. The codebase includes extensive inline documentation explaining why certain design decisions were made, making it serve double duty as both a functional application and a reference implementation of OOP principles.

Each class maintains a single, well-defined responsibility. Methods are kept short and focused on one task. Dependencies are injected rather than hard-coded, facilitating testing and future refactoring. These practices reflect professional software engineering standards applied to an educational context.

Technical Stack

  • Python and Java implementations demonstrating OOP principles in different languages
  • Class inheritance hierarchies and abstract base classes
  • Design patterns including Factory, Strategy, Template Method, and Observer
  • JSON-based data persistence for question banks
  • Modular architecture with clear separation between model, controller, and view

Student Study Pattern Analyzer

Data Analysis Tool 2025

Overview

A data analysis system that processes student study logs to identify patterns, trends, and insights. The tool accepts structured data files containing study session records—including subjects, durations, times, and performance metrics—and generates analytical reports highlighting productivity patterns, subject-wise time allocation, and temporal trends in study behavior.

The project demonstrates practical data processing skills including file parsing, data cleaning, statistical analysis, and insight generation. It transforms raw activity logs into actionable information that students can use to optimize their study strategies based on empirical evidence rather than subjective impressions.

Data Tracking and Structure

Study sessions are recorded as structured data entries with standardized fields: timestamp (date and time of session), subject identifier, duration in minutes, session type (reading, problem-solving, review), and optional performance metrics (problems completed, accuracy percentage, comprehension self-rating). This structured format enables consistent parsing and analysis across arbitrary time periods.

Data can be captured manually through a simple logging interface or automatically via integration with time-tracking applications. The system accepts both CSV and JSON input formats, with validation logic that checks for required fields, verifies data types, and flags anomalous entries (negative durations, future timestamps, invalid subject codes) for correction.

Historical data accumulates over time, enabling longitudinal analysis. The system maintains data integrity through append-only logging—new sessions are added to the dataset without modifying existing entries, preserving a complete audit trail of study activity.

Pattern Identification Logic

The analysis engine implements several pattern detection algorithms. Temporal clustering identifies preferred study times by grouping sessions into time-of-day buckets (morning, afternoon, evening, night) and calculating total duration and average session length for each period. This reveals whether a student is naturally more productive at certain times, information that can inform schedule optimization.

Subject balance analysis computes the distribution of study time across different subjects. Visualization as a proportional breakdown highlights whether time allocation aligns with stated priorities or if certain subjects are being neglected. The system can flag significant imbalances—such as spending 80% of time on one subject while three others receive minimal attention—prompting rebalancing.

Streak detection tracks consecutive days of study activity, identifying both sustained study periods and gaps. Long streaks indicate consistency, an important factor in long-term retention. Gap detection highlights periods of inactivity, which often correlate with decreased performance when study resumes. This metric encourages regular, distributed practice over cramming.

Performance correlation analysis examines relationships between session characteristics (duration, time of day, session type) and performance metrics when available. For instance, the system might identify that problem-solving sessions in the morning yield higher accuracy than those late at night, suggesting optimal scheduling for intensive work.

Statistical Processing

The system computes summary statistics including total study time, average session duration, median session length, and standard deviation. These aggregate metrics provide context for individual session evaluation—a single long session may be unremarkable if average sessions are lengthy, but noteworthy if typical sessions are brief.

Trend analysis applies moving averages to smooth day-to-day variability and reveal longer-term patterns. A 7-day moving average of total daily study time shows whether overall study volume is increasing, decreasing, or stable. Comparing recent averages against historical baselines quantifies changes in study behavior.

Percentile calculations identify outlier sessions. Sessions in the top 10% by duration might represent exceptional focus periods worth analyzing for replication. Sessions with unusually low performance despite adequate duration might indicate external factors affecting comprehension, prompting investigation into environmental or physiological conditions.

Insight Generation

Beyond raw statistics, the system generates natural language insights that interpret the data. These insights follow conditional logic: if study time for a subject has decreased by more than 30% over two weeks, generate a warning about potential neglect. If performance metrics show consistent improvement in a subject, acknowledge the positive trend and encourage continuation.

Comparative insights contextualize individual metrics against broader patterns. For example, "Your average morning session is 45 minutes, 50% longer than your evening sessions, and morning sessions show 15% higher problem accuracy." This comparison provides specific, actionable information rather than isolated statistics.

Recommendation generation suggests concrete actions based on detected patterns. If analysis shows consistent study gaps on weekends, the system might recommend scheduling specific weekend time blocks. If certain subjects consistently receive less attention, it can propose reallocation targets to achieve more balanced coverage.

Data Visualization

Results are presented through multiple visualization modes. Time-series charts display study duration over time, with subject-specific series showing how attention allocation evolves. Bar charts compare total time across subjects. Heatmaps visualize study activity across days and time blocks, revealing scheduling patterns at a glance.

Visualizations emphasize clarity over aesthetics. Axes are clearly labeled with appropriate scales. Color coding is consistent and accessible (avoiding red-green combinations that are problematic for colorblind users). Legends explain all symbols and abbreviations used.

Architecture and Extensibility

The codebase separates data ingestion, analysis, and presentation into distinct modules. The ingestion layer handles file I/O and parsing, producing standardized internal data structures. The analysis layer operates on these structures, implementing statistical calculations and pattern detection. The presentation layer formats results for output without performing calculations itself.

This modular design facilitates extension—adding new analysis algorithms requires implementing a single analysis module that conforms to the existing interface. New data sources can be integrated by writing additional parsers that produce the standard internal format. Output formats can be added through new presentation modules without modifying core logic.

Technical Stack

  • Python for data processing and statistical analysis
  • CSV and JSON parsing with schema validation
  • Statistical algorithms including moving averages, percentile calculations, and correlation analysis
  • Data visualization through charting libraries
  • Modular architecture supporting multiple input and output formats
